# career · 项目协作规范

> 本文件是本项目的主约束源。`CLAUDE.md` 仅负责把协作者导向这里。  
> 父目录 `E:\职业生涯\CLAUDE.md` 的通用规则继续生效；本文件只做项目级补充，不做削弱。

## 读取顺序

1. 开始任何工作前，先以 UTF-8 编码读取本文件。
2. 如果任务涉及创建或修改 UI、布局、样式、交互、动效、视觉文案呈现，必须先以 UTF-8 编码读取项目根目录的 `DESIGN.md`，再动手实现。
3. 如果任务涉及事实内容更新，先修改 `../公司经历/*.md`，再更新本项目数据或页面。
4. 如果规则需要调整，先改文档，再改代码或内容产物。

## 文件职责

- `AGENTS.md`：项目目标、内容模型、工程边界、协作流程、性能约束。
- `DESIGN.md`：视觉身份、设计 token、排版规则、组件视觉语义、设计禁忌。
- `UI_DESIGN.md`：具体页面结构、响应式规则、卡片状态与内容映射。
- `design-preview.html`：静态视觉原型，只用于确认 UI 方向，不作为正式工程实现。
- `CLAUDE.md`：入口说明，只负责引导读取 `AGENTS.md` 和 `DESIGN.md`。

发生冲突时按以下优先级处理：

1. 事实真实性与数据来源：`AGENTS.md`
2. 视觉和交互表现：`DESIGN.md`
3. 对外简介与使用说明：`README.md`

## 项目目标

把 `../公司经历/` 下的原始材料，变成一个一眼看懂、可以持续扩充的职业时间线页面。

- 读者：潜在合作方、招聘方、同行。
- 阅读路径：沿时间线查看公司阶段，再进入具体年份看当年做了什么、产出了什么。
- 页面不是传统简历，不追求信息量堆满；追求可信、可扫读、可外链分享。
- 正式简历负责精简履历，职业生涯页面只负责展开职业时间线，不放额外身份区、指标区或营销式介绍。

## 核心原则

1. 简洁优先。能删的就删，不因为“已经写了”而保留无效内容。
2. 可信优先。数据不明确就写定性表述，不编数字，不造结果。
3. 扫读优先。默认让访客在 30 秒内建立整体判断，细节按需展开。
4. 一致优先。每家公司使用同一信息框架，不因为某段经历重要就单独发明结构。
5. 取舍优先。简历里省略的旧技术、旧项目细节，可以放到页面里，但页面依然不能变成资料堆。

## 数据源与事实约束

- 页面内容只从 `../公司经历/*.md` 派生，本目录不另起事实源。
- 公司全称、时间格式、“至今”的处理规则，遵循父目录 `E:\职业生涯\CLAUDE.md`。
- 量化数据必须能追溯到公司经历文档；追溯不到时，一律改成定性描述。
- 每段经历在代码或数据层必须能反向定位到源 Markdown 文件。
- 简历版内容可以做压缩，但职业生涯页面不能通过“润色”改变事实边界。

## 内容模型

页面只展示职业时间线。内容层级固定为：

```text
公司阶段
└── 年份事件
    ├── 做了什么
    └── 产出 / 价值
```

公司阶段字段：

| # | 字段 | 含义 | 长度上限 | 缺失处理 |
| - | ---- | ---- | -------- | -------- |
| 1 | 公司 | 公司全称 | 一行 | 不允许缺失 |
| 2 | 时间 | `YYYY.MM – YYYY.MM`，在职写 `至今` | 一行 | 不允许缺失 |
| 3 | 阶段定位 | 这一家公司阶段的角色变化或业务定位 | 一句 | `⟨待补充⟩` |

年份事件字段：

| # | 字段 | 含义 | 长度上限 | 缺失处理 |
| - | ---- | ---- | -------- | -------- |
| 1 | 年份 | `YYYY` 或 `YYYY.MM` | 一行 | 不允许缺失 |
| 2 | 主题 | 当年最重要的一句话概括 | 一句 | `⟨待补充⟩` |
| 3 | 做了什么 | 主要项目、职责、动作 | `<= 3` 条 | `⟨待补充⟩` |
| 4 | 产出 / 价值 | 可被他人感知的结果、沉淀、业务价值 | `<= 3` 条 | `⟨待补充⟩` |

写作规则：

- 年份事件可持续追加，新事件只新增一个 year block，不改既有结构。
- 能写年份就写年份；年份不确定时写时间段，并标注为待核实。
- “做了什么”写动作，“产出 / 价值”写结果，不混在一起。
- 技术栈只在帮助理解问题复杂度时出现，不单独堆成标签墙。

## 页面结构

页面结构保持克制，只保留职业时间线：

1. 时间线标题：仅说明这是职业时间线
2. 公司阶段：倒序展示公司
3. 年份事件：每家公司内部按倒序展示年份事件
4. 事件内容：每条事件展示“做了什么”和“产出 / 价值”

硬约束：

- 时间线必须倒序，最近经历在最上方。
- 页面不展示身份头、关键指标、联系方式页脚、简历入口等非时间线内容。
- 公司阶段和年份事件都要支持 URL 锚点，方便简历或外链指向具体位置。
- 不弹 Modal，不做订阅框，不做打断式引导。

## UI 与交互约束

与视觉相关的具体实现以 `DESIGN.md` 为准，这里只定义硬约束：

- 移动优先，先保证 360px 宽可读，再扩展桌面布局。
- 键盘可达：卡片可 Tab 聚焦，Enter / Space 可展开收起。
- 反馈即时：展开 / 收起、悬停 / 聚焦反馈必须在 200ms 左右完成。
- 默认静态优先：无动效也要成立；有动效只能服务于层级和视线引导。
- 页面必须像可信的编辑性文档，不像营销站、活动页或管理后台。

## 内容呈现策略

- 简历负责“精简履历”，页面负责“展开职业时间线”。
- 过时技术、历史上下文、被简历压缩掉的细节可以放到对应年份事件里，不能单独开板块。
- 页面优先展示持续有效的能力：平台化能力、系统抽象能力、规模化交付能力、团队杠杆、跨端复杂度处理能力。
- 技术栈只在帮助理解问题复杂度时出现，不做成密集标签墙。

## 性能约束

- 首屏可交互目标：`<= 3s`（4G 网络基准）。
- 首屏 JS 预算：`<= 50KB`（gzip）；超出必须在 `## 依赖决策` 说明原因。
- 图片默认懒加载，且尺寸必须匹配容器。
- 不引入阻塞渲染的第三方脚本，包括埋点、客服、外部字体 CDN。
- 字体必须本地化自托管或确保 `font-display: swap`。

## 技术约束

- 静态优先：优先选择纯静态 HTML 或 SSG；如果引入 SSR / 服务端能力，必须先记录理由。
- 组件自写优先；UI 库、动效库、重状态管理默认不引入。
- 数据流固定为：`公司经历/*.md -> 转换脚本 -> content/ -> 页面`。
- `content/` 只能由脚本生成，禁止手工修改。
- 代码、变量名、命令用英文；正文和注释用中文。

## 目录结构

```text
career/
├── AGENTS.md           项目主约束
├── CLAUDE.md           入口文件
├── DESIGN.md           视觉身份与设计 token
├── UI_DESIGN.md        具体 UI 设计稿
├── design-preview.html 静态视觉原型
├── README.md           对外简介
├── src/                页面源码
├── content/            结构化数据产物（脚本生成，勿手改）

└── public/             字体、favicon、静态资源
```

新建其他顶层目录前，必须先在本文件登记用途。

## 协作与变更流程

1. 新增或修改一段经历：先改 `../公司经历/公司全称.md`，再跑转换脚本，不要直接改 `content/`。
2. 调整视觉或交互：先读并修改 `DESIGN.md`，再修改页面实现。
3. 调整项目行为或结构：先改本文件，再改实现。
4. 提交前至少完成一次可用性验证：构建成功、内容可读、移动端不炸版、锚点可访问。

## 反模式

- 给“特别重要”的公司新增专属结构或额外字段。
- 用 emoji、装饰图标、贴纸、插画代替信息表达。
- 编造不确定的量化结果。
- 为了好看添加与阅读无关的动画，如粒子、视差、打字机效果。
- 在页面中塞联系表单、订阅框、客服机器人、Cookie 同意条等打断式元素。
- 直接手改结构化数据产物。
- 未读取 `DESIGN.md` 就直接改 UI。
